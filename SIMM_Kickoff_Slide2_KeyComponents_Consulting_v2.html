Key Components of SIMM Model Validation

Phase 2 Content Page | High-density consulting draft | Minimum font size = 11-12 range (optimize for one-page briefing print)
Validation mission: Prove Internal System B’s SIMM implementation is technically correct, data-complete, and regulatorily defensible under ISDA SIMM v2.8+2506 + HKMA CR-G-14. Evidence standard: requirement-to-test traceability with reproducible calculation artifacts and documented root-cause analysis for all material variances.

1) Scope Design & Testing Logic

  • Coverage rule: one representative trade per risk type, stratified by product family, currency, and option moneyness (ATM/ITM/OTM), then expanded where materiality requires.
  • Risk stack: Delta, Vega, Curvature, and Base Correlation (Credit Qualifying only), including concentration scaling impact.
  • Portfolio layers: sampled deep-dive (design defects) + full-population controls (production integrity).
  • Control objects: versioning, parameter pack, risk mapping, aggregation logic, sensitivity production, data lineage, reconciliation and governance evidence.
  • Out-of-scope: independent curve/surface reconstruction (unless separately mandated); scope is consistency and mapping validation between valuation and SIMM.
  • Materiality policy: set variance thresholds by component (risk-class IM, product-class IM, total IM), then trigger RCA workflow for breaches.
  • Three assurance dimensions: regulatory compliance, computational correctness, and data governance sustainability.

Critical Evidence Required

  • Version memo (methodology + effective date + parameter release)
  • Sensitivity extracts with mapping keys and timestamp lineage
  • Aggregation intermediate outputs (bucket-level and risk-class level)
  • GL/Sub-ledger/Golden-source reconciliation snapshots
  • Independent recompute logs and exception register
2
Methodology Implementation + Sensitivity Verification
A. SIMM Methodology Controls
  • Confirm SIMM version in production aligns with policy baseline.
  • Validate risk weights/correlation matrices/concentration thresholds are loaded correctly and date-effective.
  • Check product-class and risk-class aggregation flow: no unintended netting or diversification outside the spec.
  • Verify Base Correlation treatment appears only where applicable and is not double-counted with spread risk.
  • Reconcile intermediate K/S terms for selected buckets to detect implementation drift.
B. Sensitivity Production Controls
  • Delta/Vega/Curvature generation aligns with pricing model conventions and shock definitions.
  • Test bump-size, optionality capture, sign conventions, and unit normalization.
  • Validate mapping keys: currency, tenor, issuer, bucket, sector, product class.
  • Challenge representative trades independently; compare component sensitivities before IM aggregation.
  • Escalate if mismatch indicates model, mapping, or data-engine interface defects.
3
Data Quality, Independent Recompute, and Outcome Analysis
C. Data Integrity + Reconciliation
  • Sample-level checks: position completeness, reference mapping, market data timeliness.
  • Full-population checks: IM input/output reconciliation to GL, sub-ledger, and golden source.
  • Curve/surface consistency: identifier, timestamp, and usage alignment between valuation and SIMM stack.
  • Stale/missing data controls: fallback rules, override approvals, and exception journaling.
  • Lineage proof: source → transformation → sensitivity → IM result trace.
D. Independent Recompute + Backtesting Review
  • Run independent SIMM engine and compare Delta/Vega/Curvature/risk-class/total IM levels.
  • Apply tolerance grid and root-cause tree (parameter, mapping, aggregation, or source data).
  • Review 1-day, 10-day, and rolling backtesting methodology design against latest requirements.
  • Document findings as compliant / partial / gap with remediation owner and retest criteria.
  • Produce regulator-ready pack: methods, evidence, exceptions, and closure status.
